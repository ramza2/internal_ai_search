# 백그라운드 작업 큐/워커 구조 설계

> **문서 목적:** internal-ai-search 프로젝트에서 인덱싱·동기화 등 장시간 작업을 HTTP 요청 밖으로 분리하기 위한 **향후 구현 가이드**이다.  
> **범위:** 설계만 수행한다. 본 문서 작성 시점에는 worker 구현, DB 마이그레이션, API/프론트 변경을 하지 않는다.

---

## 1. 배경

### 1.1 현재 구조

- 관리자가 프론트엔드에서 단계별 또는 파이프라인 자동 실행 버튼을 클릭한다.
- FastAPI가 해당 요청을 받아 **동일 HTTP 요청 컨텍스트 안에서** WebDAV 탐색, 파일 다운로드, 파싱, chunk/embedding 처리 등을 수행한다.
- 클라이언트(브라우저)는 작업이 끝날 때까지 **응답을 대기**한다.

### 1.2 현재 동기 작업 API

다음 엔드포인트는 (설계 시점 기준) 장시간 작업이 HTTP 요청 내에서 완료되는 형태로 사용된다.

| API | 역할 |
|-----|------|
| `POST /api/data-sources/{id}/sync-tree` | WebDAV 재귀 동기화 |
| `POST /api/data-sources/{id}/process-pending-text` | 일반 텍스트 처리 |
| `POST /api/data-sources/{id}/process-pending-documents` | 문서 파일 처리 |
| `POST /api/data-sources/{id}/chunk-completed-text` | Chunk 생성 |
| `POST /api/data-sources/{id}/embed-pending-chunks` | Embedding 생성 |

### 1.3 문제점

- **Timeout:** 프록시·로드밸런서·브라우저·Uvicorn 등에서 장시간 요청이 끊길 수 있다.
- **상태 추적:** 사용자가 새로고침하거나 탭을 닫으면 진행률·남은 작업을 알기 어렵다.
- **서버 재시작:** 진행 중인 작업이 요청 스레드에 묶여 있으면 복구·재개가 어렵다.
- **부하 집중:** 대량 파일 처리 시 메모리·DB 연결·WebDAV·임베딩 API에 단기간 부하가 몰릴 수 있다.
- **운영 기능:** 취소, 재시도, 우선순위, 진행률(퍼센트) 표시를 동기 모델만으로 일관되게 제공하기 어렵다.

---

## 2. 목표

백그라운드 작업 구조를 도입했을 때의 목표는 다음과 같다.

- **짧은 API 응답:** 작업 생성 요청은 `job_id` 등 식별자를 반환하고 즉시 종료한다.
- **Worker 처리:** 실제 I/O·CPU 집약 작업은 별도 worker 프로세스(또는 스레드 풀과 분리된 실행 단위)에서 수행한다.
- **진행률 조회:** 프론트는 `job_id`로 상태·진행률을 polling(또는 향후 SSE/WebSocket)한다.
- **이력:** 성공·실패·취소·재시도를 job 단위로 남긴다.
- **기존 자산과의 정합:** `scan_jobs` 및 `action_logs`를 가능한 한 일관된 모델로 활용한다.
- **운영 안전성:** 대량 작업을 타임아웃 없이 실행하고, 운영자가 상태를 파악·중단할 수 있게 한다.

## 3. 대상 작업

### 3.1 1순위 (인덱싱 파이프라인 핵심)

1. WebDAV 재귀 동기화 (`sync-tree`)
2. 일반 텍스트 파일 처리 (`process-pending-text`)
3. 문서 파일 처리 (`process-pending-documents`)
4. Chunk 생성 (`chunk-completed-text`)
5. Embedding 생성 (`embed-pending-chunks`)

### 3.2 2순위

- 위 5단계를 **한 번에 큐에 올리는** “전체 파이프라인” job (부모 job + 단계별 child job 또는 단일 job 내 상태 머신)
- `detect_deleted` 등 **삭제 감지 포함** sync-tree 변형·대용량 옵션
- `reprocess` / `reembed` 등 **재처리** 성격의 배치

### 3.3 3순위

- 중복·유사 문서 탐지
- 오래된 로그·파일 정리(아카이브/삭제 정책)
- 향후 OCR 파이프라인
- 향후 HWP/DOC/XLS/PPT 등 추가 포맷 변환·처리

## 4. 추천 아키텍처

### 4.1 권장 데이터 흐름

```
[Admin UI]
        │  작업 생성 요청 (짧은 HTTP)
        ▼
[FastAPI  —  예: POST /api/admin/jobs]
        │  job 레코드 생성 (PENDING)
        ▼
[PostgreSQL  —  scan_jobs 또는 jobs 테이블]
        │  대기 큐
        ▼
[Job Queue  —  논리적 큐; 구현은 DB polling 또는 Redis 등]
        │  worker가 dequeue
        ▼
[Worker Process]
        │  기존 service 레이어 재사용
        ▼
[PostgreSQL  —  files / file_contents / document_chunks 등 갱신]
        │
        ▼
[Frontend  —  GET /api/admin/jobs/{job_id} 폴링]
```

### 4.2 텍스트 다이어그램

```
[Admin UI]
   ↓ 작업 생성 요청
[FastAPI /api/admin/jobs]
   ↓ job 생성
[PostgreSQL scan_jobs]
   ↓ 대기 상태 (PENDING)
[Worker]
   ↓ 작업 실행
[files / file_contents / document_chunks 업데이트]
   ↓
[Frontend Polling /api/admin/jobs/{job_id}]
```

## 5. 선택 가능한 큐 방식 비교

### A. FastAPI `BackgroundTasks`

| 장점 | 단점 |
|------|------|
| 구현이 가장 단순 | 프로세스 재시작 시 **작업 유실** 가능성 |
| 별도 인프라 불필요 | 장시간 작업·다중 worker와 맞지 않음 |
| 개발·데모에 빠름 | 수평 확장(다중 앱 서버) 시 큐 공유 불가 |

**적합도:** 개발/데모용 단계적 완화에만 권장. 본 프로젝트의 “운영 가능한 백그라운드” 목표에는 부족.

### B. DB Polling Worker

| 장점 | 단점 |
|------|------|
| **PostgreSQL만**으로 큐 역할 가능 | 전문 메시지 브로커 대비 처리량·지연 한계 |
| Redis 등 **추가 인프라 없이** 운영 가능 | `FOR UPDATE SKIP LOCKED`, 중복 방지 등 **설계 필요** |
| 기존 **`scan_jobs`와 자연스럽게 정합** | 폴링 간격에 따른 지연 |

**적합도:** 현재 스택(MVP → 소규모 운영)에 **가장 덜 침습적**이며, 1차 권장안.

### C. Redis Queue / RQ

| 장점 | 단점 |
|------|------|
| Python에서 단순한 API | **Redis 운영** 필요 |
| worker 분리·재시도 패턴 명확 | 인프라·모니터링 범위 증가 |

**적합도:** MVP 이후, 트래픽·작업량 증가 시 검토.

### D. Celery + Redis / RabbitMQ

| 장점 | 단점 |
|------|------|
| 생태계·모니터링·스케줄링 풍부 | **설정·운영 복잡도** 높음 |
| 대규모에 적합 | 현재 규모 대비 과할 수 있음 |

**적합도:** 대규모 운영·다수 팀이 작업 큐를 표준으로 쓸 때.

### 결론

**초기 운영 전환 단계에서는 “DB Polling Worker”를 1차 권장안**으로 둔다.

- 이미 PostgreSQL을 사용 중이다.
- `scan_jobs` 테이블이 존재해 job 메타데이터·진행 카운트와의 매핑이 자연스럽다.
- Redis 없이도 소규모 배포(Linux 단일/소수 서버)를 단순하게 유지할 수 있다.
- 기존 동기 API를 한 번에 대체하지 않고 **병행 도입**하기 좋다.

## 6. `scan_jobs` 활용 방안

### 6.1 현재 활용 예시 (필드)

설계 문서 수준에서 일반적으로 기대되는 필드 예시는 다음과 같다 (실제 스키마는 코드·마이그레이션을 기준으로 조정).

- `id`
- `data_source_id`
- `job_type`
- `status`
- `started_at`, `finished_at`
- `total_files`, `processed_files`, `completed_files`, `failed_files`, `skipped_files`, `deleted_files`
- `error_message`, `current_file_path`

### 6.2 추가 검토 필드 (DB 변경은 향후 마이그레이션에서 결정)

| 필드 | 용도 |
|------|------|
| `requested_by` | job 요청자(user id); 감사 추적 |
| `job_params` (JSONB) | sync depth, limit, 확장자 필터 등 (**비밀번호·원문 저장 금지**) |
| `progress_percent` | UI용; 없으면 카운트 기반으로 계산 가능 |
| `cancel_requested` | 취소 요청 플래그 |
| `worker_id` | 어떤 worker 인스턴스가 잡았는지 |
| `heartbeat_at` | stale 판별 |
| `parent_job_id` | 파이프라인 부모 job |
| `pipeline_step` | 현재 단계(enum 또는 문자열) |
| `retry_count` / `max_retries` | 재시도 정책 |

**주의:** 본 문서 단계에서는 **실제 DB 컬럼 추가를 하지 않는다.** 위는 구현 전 검토용 후보뿐이다.

## 7. Job 상태 정의

### 7.1 제안 상태 집합

| 상태 | 의미 |
|------|------|
| `PENDING` | 큐에 등록, worker 미할당 |
| `RUNNING` | worker가 실행 중 |
| `COMPLETED` | 정상 완료 |
| `FAILED` | 복구 불가 또는 정책상 실패 종료 |
| `CANCELLED` | 사용자 취소 또는 정책상 중단 완료 |
| `CANCELLING` | 취소 요청 수신, worker가 안전 지점에서 종료 중 |
| `PARTIAL` | 일부 파일/배치 실패했으나 job은 종료(재시도 가능 여부는 별도 플래그로) |

### 7.2 기존 `scan_jobs.status`와의 관계

- 현재 DB에 저장된 `status` 값 집합이 위와 **완전히 일치하지 않을 수 있다.**
- 구현 시점에 다음 중 하나를 선택해야 한다.
  - **(a)** 기존 값에 매핑 테이블을 두고 API 레이어에서만 통일된 enum 노출
  - **(b)** 마이그레이션으로 값을 정규화하고 코드 전역 수정
- `PARTIAL`, `CANCELLING` 등이 **신규**라면: 마이그레이션 + 기존 row 기본값 처리 + 백필 정책이 필요하다.

### 7.3 상태 전이 예

```
PENDING → RUNNING → COMPLETED
PENDING → RUNNING → FAILED
RUNNING → CANCELLING → CANCELLED
RUNNING → PARTIAL
```

## 8. 신규 API 설계안

> 아래는 **향후 추가**를 가정한 설계안이다. 경로·필드명은 구현 시 조정 가능하다.

### 8.1 작업 생성

`POST /api/admin/jobs`

**Request 예시:**

```json
{
  "job_type": "SYNC_TREE",
  "data_source_id": "uuid",
  "params": {
    "start_path": "/",
    "max_depth": 3,
    "max_items": 5000,
    "include_hidden": false,
    "apply_exclusions": true,
    "detect_deleted": false
  }
}
```

**Response 예시:**

```json
{
  "status": "ok",
  "job_id": "uuid",
  "message": "Job queued successfully"
}
```

### 8.2 작업 조회

`GET /api/admin/jobs/{job_id}`

- 상태, 진행 카운트, `current_file_path`, `error_message`, `started_at`/`finished_at`, (있다면) `pipeline_step` 등 반환.

### 8.3 작업 목록

`GET /api/admin/jobs?status=RUNNING&data_source_id=...`

- 필터: `status`, `data_source_id`, `job_type`, 기간 등.

### 8.4 작업 취소

`POST /api/admin/jobs/{job_id}/cancel`

- `cancel_requested` 또는 상태를 `CANCELLING`으로 표시하고, worker가 인지 후 `CANCELLED`로 마무리.

### 8.5 작업 재시도

`POST /api/admin/jobs/{job_id}/retry`

- 새 job 생성 또는 동일 job의 재큐잉 정책은 구현 시 결정(멱등성·중복 방지 규칙 필요).

### 8.6 파이프라인 작업 생성

`POST /api/admin/pipeline-jobs`

**Request 예시:**

```json
{
  "data_source_id": "uuid",
  "steps": [
    "SYNC_TREE",
    "PROCESS_TEXT",
    "PROCESS_DOCUMENTS",
    "CHUNK",
    "EMBED"
  ],
  "params": {
    "sync_tree": {},
    "process_text": {},
    "process_documents": {},
    "chunk": {},
    "embed": {}
  }
}
```

- 부모 job 1건 + 단계별 child job, 또는 단일 job에 `pipeline_step` 전이 모델 중 선택.

## 9. Worker 처리 방식 (DB Polling Worker 기준)

### 9.1 기본 루프

1. worker가 주기적으로 `scan_jobs`(또는 동일 역할 테이블)에서 `status = 'PENDING'`인 행을 조회한다.
2. `SELECT ... FOR UPDATE SKIP LOCKED`로 **한 건**만 가져와 동시 실행 worker 간 충돌을 방지한다.
3. 트랜잭션 내에서 `status = 'RUNNING'`, `started_at`, `worker_id`, `heartbeat_at`을 갱신한다.
4. `job_type`에 따라 **기존 service 함수**를 호출한다 (HTTP 핸들러와 동일 로직 공유 권장).
5. 처리 중 `processed_files`, `current_file_path`, `heartbeat_at`을 일정 간격으로 갱신한다.
6. 정상 종료 시 `COMPLETED`, `finished_at` 설정.
7. 처리 불가 오류 시 `FAILED`, `error_message` 설정.
8. `cancel_requested = true`(또는 동등 신호)이면 **안전 지점**(파일 단위·배치 단위)에서 중단 후 `CANCELLED`.

### 9.2 동시성·중복 방지

- worker는 **여러 프로세스**로 늘릴 수 있게 설계한다.
- `SKIP LOCKED`는 필수에 가깝다.
- **동일 `data_source_id` + 동일 `job_type`** 의 `RUNNING`/`PENDING` 중복을 허용할지 여부는 제품 정책으로 결정한다.
  - 권장: 동일 유형의 장시간 job은 **한 건만** 허용하거나, 큐에 쌓되 앞선 job 완료 후 시작.

## 10. 기존 API와의 전환 전략

동기 API를 한 번에 제거하지 않고 단계적으로 전환한다.

| 단계 | 내용 |
|------|------|
| **1단계** | 기존 `POST .../sync-tree` 등 **동기 API 유지**. 신규 `POST /api/admin/jobs` 등 **비동기 API 추가**. 프론트 `PipelineRunModal`에 **「백그라운드 실행」** 등 병행 UI 추가. |
| **2단계** | 대량 작업 가이드로 **비동기 API 권장**. 동기 API는 소량·개발용으로 문서화. |
| **3단계** | 운영 환경에서 동기 API를 **숨기거나** 관리자 고급 옵션으로만 노출. |
| **4단계** | (선택) 오래 걸리는 동기 API는 내부적으로 job 생성 후 **202 + job_id** 또는 짧은 응답으로 변경—이때는 클라이언트 계약 변경이 크므로 신중히. |

---

## 11. 프론트엔드 전환 설계

> 본 문서 작성 단계에서는 **코드 변경 없음**. 방향만 정리한다.

- `PipelineRunModal`에서 **동기 실행** vs **백그라운드 실행**을 구분한다.
- 백그라운드 실행 시 **`job_id` 표시**, 상세는 작업 목록 화면으로 링크.
- **Polling:** `GET /api/admin/jobs/{job_id}`를 수 초 간격으로 호출(백오프·탭 비활성 시 감속 등 고려).
- **작업 목록 화면** 신설 후보:
  - `/admin/jobs`
  - `/admin/jobs/{job_id}`
- 작업 **취소·재시도** 버튼을 job 상세에 배치.
- 관리자 대시보드에 `RUNNING` / `PENDING` job 수 요약 카드 추가 검토.

## 12. 진행률 계산 방식

- `total_files`가 신뢰 가능하면: `processed_files / total_files` 기반 퍼센트.
- `total_files`를 사전에 모르는 경우(WebDAV 탐색 초기): **처리 건수·현재 경로** 중심 표시가 현실적이다.
- **파이프라인 job:** child job별 진행률의 가중 평균, 또는 단계 인덱스 / 전체 단계 수.
- **Embedding:** `processed_chunks / target_chunks` (target 산출 가능할 때).

**주의:** 초기에는 **정확한 percent보다 “처리된 항목 수 + 현재 파일 경로”** 가 운영 가치가 높다.

---

## 13. 장애/복구 전략

| 상황 | 대응 방향 |
|------|-----------|
| 서버 재시작 | job은 DB에 남김. `RUNNING`이면 **STALE** 정책으로 `FAILED` 또는 `PENDING` 재큐 |
| worker 중단 | heartbeat 미갱신 → stale 처리 |
| `RUNNING` + heartbeat 오래됨 | `JOB_STALE_TIMEOUT` 초과 시 다른 worker가 인수하거나 실패 처리 |
| WebDAV 인증 실패 | job `FAILED`, `action_logs`에 요약 기록 |
| 파일 하나 실패 | 파일 단위 로그·`scan_failures` 등; job 전체는 `PARTIAL` 가능 |
| embedding 모델/외부 API 다운 | 재시도 가능이면 `retry_count`; 아니면 `FAILED` |
| DB 일시 오류 | 트랜잭션 재시도·백오프; 지속 시 `FAILED` |

**취소:** 배치/파일 단위 **안전 지점**에서만 중단해 DB 불일치를 최소화한다.

## 14. 운영 설정 (환경 변수 후보)

| 변수 | 예시 | 설명 |
|------|------|------|
| `WORKER_ENABLED` | `false` | worker 프로세스 기동 여부 |

## 15. 보안/권한

- Job **생성·취소·재시도**는 **ADMIN**만 가능하도록 한다.
- Job **조회**도 **ADMIN**으로 제한하는 것이 기본안(감사·부하 관점).
- `action_logs`에 job 생성·취소·재시도·(선택) 상태 전이 요약 기록.
- `job_params`에 **credential·파일 본문·LLM prompt** 저장 금지. 필요 시 `data_source_id`만 저장하고 자격 증명은 서버 측 기존 저장소에서 조회.

## 16. 구현 우선순위 제안

1. `scan_jobs` 기반 **job status 조회 API** (목록·단건)
2. **DB polling worker 스켈레톤** (dequeue + heartbeat + stale)
3. **sync-tree** 비동기화(첫 번째 장시간 후보)
4. **process-pending-text** 비동기화
5. **process-pending-documents** 비동기화
6. **chunk** 비동기화
7. **embedding** 비동기화
8. **pipeline job** (부모·단계 모델 확정 후)
9. 프론트 **`/admin/jobs`** 화면
10. **`PipelineRunModal`** 백그라운드 실행 연동
