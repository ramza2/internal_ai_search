# 전체 저장소 처리 모드 (scan_scope=FULL)

## 목적

운영자가 저장소 화면 **검색 반영**을 눌렀을 때 기본 동작은 “일부 테스트”가 아니라 **저장소 전체**를 AI 검색에 반영하는 것이다. 다만 무제한 동기 HTTP 실행은 허용하지 않는다.

## 정책 요약

| 구분 | `scan_scope` | `max_depth` / `max_items` | 실행 경로 |
|------|----------------|---------------------------|-----------|
| 전체 저장소 | `FULL` | 무시 (`null` 저장 가능) | Worker / `PIPELINE` / `POST /api/admin/jobs/sync-tree` |
| 제한 수집 | `LIMITED` (기본) | 기존과 동일 (기본 3 / 5000, 상한 20 / 50000) | 위 + **동기** `POST .../sync-tree` |
| 동기 API + FULL | — | — | **400** — “백그라운드 파이프라인에서 실행” |

## 백엔드

- `app/services/sync_tree_scope.py`: `FULL` / `LIMITED` 정규화, effective depth/item 해석.
- `FULL`: 내부적으로 매우 큰 depth/item 상한 사용. 선택 설정 `SYNC_TREE_EMERGENCY_MAX_ITEMS` (>0) 시 항목 수 emergency cap.
- `collect_tree`는 여전히 BFS 후 **메모리에 전체 목록**을 모은 뒤 upsert (대규모 저장소 메모리 이슈 → 후속 streaming upsert TODO).
- Worker: `cancel_check` / heartbeat 유지 (`file_recursive_sync_service`).
- `default_pipeline_params().sync_tree`: `FULL`, `detect_deleted=true`, 파일 처리 `max_file_size_bytes` = 서버 상한 **256MB**.

## 프론트

- `PipelineRunModal`: 기본 `scan_scope=FULL`, CTA **전체 검색 반영 작업 등록**, 깊이/항목 “제한 없음”은 FULL일 때만 표시.
- 파일 크기: **서버 허용 최대 256MB** (진짜 무제한 표현 금지).
- `AdminJobsEnqueuePanel`: sync-tree 등록 시 FULL/LIMITED 선택.

## 운영 참고

- 첫 전체 반영은 오래 걸릴 수 있음. 진행·취소는 **작업 이력** / worker.
- `detect_deleted=true` 는 대량 저장소에서 DB 비용이 클 수 있음.
- 향후: streaming traversal/upsert, 대규모 진행률 추정, 저장소별 총 파일 수 사전 계산.
